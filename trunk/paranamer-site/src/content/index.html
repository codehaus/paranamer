Method Parameter Name Access for versions of Java PRIOR TO Java 7.0  

What is it ???

It is a mechamism for accessing the parameter names of methods of Java classes compiled into jars. Normally the javac compiler drops parameter names making a method like 'doSometing(mypkg.Person toMe)' look like 'doSomething(mypkg.Person arg0)' to people using Java's reflection to inspect methods.  To date this has not been very useful to Java application developers, but with the advent of advanced scripting languages and web action frameworks for the JVM it is of increasing importance to be able to leverage a method's parameter names. Scripting languages like Groovy & JRuby, web action frameworks like Waffle and VRaptor (that verge on the tranparent) and the compelling Grails.

ParaNamer allows you to generate and use parameter name info for versions of Java prior to Java 7.0 (when Java may get parameter name access). Generation and use are two quite separate things.

Quick Start

Generating Parameter Name Meta-data with Ant

Once you have downloaded the ParaNamer jar, in your Ant script do the following after <javac/>:

<paranamer sourceDirectory="src/java" outputDirectory="target/classes"/>

Then, when you're making your jar, ensure that META-INF/ParameterNames.txt does not get omitted

Download ParaNamer from here - http://dist.codehaus.org/paranamer/
ParaNamer needs a snapshot of qdox-20060801.022451.jar (which is a snapshot release) or qdox-1.6.jar (or above) while generating parameter info.

Generating Parameter Name Meta-data with Maven 2

For Maven, it is simpler. Just add this to the <build><plugins> section of your pom.xml:

<plugin>
    <groupId>com.thoughtworks.paranamer</groupId>
    <artifactId>paranamer</artifactId>
    <version>0.9-SNAPSHOT</version>
    <executions>
        <execution>
            <id>run</id>  <!-- id is optional -->
            <configuration>
                <sourceDirectory>${basedir}/src/java</sourceDirectory>
                <outputDirectory>${basedir}/target/classes</outputDirectory>
            </configuration>        
            <goals>
                <goal>run</goal>
            </goals>
        </execution>
    </executions>
</plugin>

The jar file will automatically be made with parameter name meta-data.

Using Paranamer

If you are leverging the paranamer Jar at runtime, there is a method called 'lookup' that does the business:

MySomethingOrOther.java
  Method method = new DefaultParanamer().lookupMethod(AClass.class.getClassLoader(), 
              "foo.BarAction", "addToShoppingCart", "make,model,qty,onClick");

It returns null if there are is no method that matches what you're looking for.  Alternatively, there are similar methods  'checkedLookup' and 'uncheckedLookup' that throw checked or unchecked exceptions when the methods can't be found for any reason. They are in adjacent classes CheckedParanamer and UncheckedParanamer. Its your preference as to which to use.

The unit tests for ParaNamer illustrate some of the ways of using it, if you're interested, look here -> http://svn.paranamer.codehaus.org/browse/~raw,r=9999/paranamer/trunk/src/test/com/thoughtworks/paranamer/ParanamerExampleTestCase.java

ParaNamer does not have any runtime jar dependecies while looking up parameter info previously generated that's been zipped into a jar.

Using Paranamer without using Paranamer's jar

There are already too many jar's for day to day Java development right?  Simply copy the lookupMethod method to your codebase. All the code is public domain. People are free to do what they like to it, copy portions into your GPL or Commercial or Apache licensed project and retain GPL, Commercial and Apache licenses (or whatever).

A link to the source to copy is here - http://svn.paranamer.codehaus.org/browse/~raw,r=9999/paranamer/trunk/src/java/com/thoughtworks/paranamer/DefaultParanamer.java

You may also want some of the functionalty from the Caching variant - http://svn.paranamer.codehaus.org/browse/~raw,r=9999/paranamer/trunk/src/java/com/thoughtworks/paranamer/CachingParanamer.java

Using Paranamer with ASM

If generating meta information on parameter names at compile time is not for you, try Paranamer with ASM as a runtime only solution. ASM allows java code to parse class files, and is useful for Paranamer if debug data is still compiled into the classes.

MySomethingOrOther.java
  Method method = new AsmParanamer().lookupMethod(AClass.class.getClassLoader(), 
              "foo.BarAction", "addToShoppingCart", "make,model,qty,onClick");

How Paranamer works

It creates a file in META-INF/ called ParameterNames.list  - Yes its going to add to jar size.

packagename.ClassName methodName paramName0,paramName1 Class0,Class1
packagename.ClassName anotherMethodName paramName0 Class0

Clearly the method's source is analysed and lines are added per method to the ParameterNames.list file.  "What if a parameter name(s) changes? That's a real possibility. Rather than have to worry about a hard choice of breaking backwards compatibility versus never change your parameter names after shipping your product, there's a reasonable way of allowing changes to parameter names going forward, and still supporting the previous parameter names.  Add a standard doclet tag to illustrate the prior name, which ParaNamer also copies into ParameterNames.list:

/**
 * Blah blah blah blah
 *
 * @previousParamNames d,m,y
 */
public void setDateOfBirth(int day, int month, int year) { ... }

ParaNamer will go away in the future when Java gets Reflection access to Method parameter names. That is likely to look in the worst case like:

Method method = Foo.class.getMethod(....)
String[] parameterNames = method.getParameterNames(); // throws ?? or returns null ???

Though there could be some convenience finders like for Paranamer:

Foo.class.getMethodForParameters("methodName", new String[] {"paramName0","paramName1"});
Foo.class.getMethodForParameters("methodName", "paramName0","paramName1"); // varargs
